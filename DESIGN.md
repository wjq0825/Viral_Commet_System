# 神评论系统设计文档 (Final Version)

## 1. 初始假设 (Hypotheses)
- **H1: 模式胜于内容**。神评论的魅力往往在于其"结构"（如：神反转、降维打击、共情替身），而非具体的文字。
- **H2: 多模态是灵魂**。在图文帖中，神评论通常是文字对图片的"二次解构"。只读文字会丢失 70% 的上下文。
- **H3: 风险在于边界**。最高级的幽默往往游走在冒犯的边缘，系统必须有程序化的"硬隔离"来确保不越界。

## 2. 验证与反证 (Verification & Falsification)
- **正向验证**：通过 10-20 条涵盖不同平台、不同语气的帖子，验证系统是否能识别出每条帖子的"Hook Points"。
- **反证 (边界测试)**：
    - 输入完全无关的图文（如：严肃财报配一张可爱的猫图），观察系统是否会强行寻找联系。
    - 注入恶意指令（Prompt Injection），验证 Hook 是否能成功拦截。
    - 模拟图片加载失败，验证系统是否能优雅降级而不是编造内容。

## 3. 问题重构 (Problem Reframing)
我们将"生成神评论"重构为：**"在特定上下文下的模式匹配与创意重构"**。
系统不应被视为一个"文案写手"，而应被视为一个**"语境解码器 + 成功套路应用引擎"**。

### 4.2 Schema设计思路与字段分层

#### 为什么这样分层？

1. **Factual Fields（事实字段）** - 客观可验证
   - 不会因为模型主观性而改变
   - 便于检索引擎的准确性（如：根据title和platform检索相关Reddit评论）
   - 允许人工审核时快速溯源到原始帖子

2. **Judgement Fields（判断字段）** - 模型主观生成
   - 需要多模态理解（文字+图片+语气+文化背景）
   - 这些判断直接指导后续的"模式学习"和"评论生成"
   - 风险评估至关重要，必须分层评估而不是简单的1-10评分

#### 为什么要强调"图文双维度"？

- **纯文本理解的盲点**：一条仅有文字的帖子可能显示"严肃"，但如果配的是一张滑稽的图片，实际语气就变成了"讽刺"。
- **图片决定梗点**：许多爆款帖子的笑点在于"图片与文字的反差"。系统必须显性地分析这种反差。
- **跨模态的互动引导**：最优的神评论往往是针对"图片内容+文字表达"的综合反应，而非仅针对文字。

#### 什么是"事实字段"vs"判断字段"？

| 字段 | 类型 | 示例 | 是否需要AI |
|---|---|---|---|
| `id`, `title`, `content` | 事实 | "Trump says Israel..." | 否 |
| `platform`, `timestamp` | 事实 | "Reddit", "2026-05-13" | 否 |
| `images.description` | 事实* | "A cat in a car looking angry" | 是（需要图像识别） |
| `core_topic` | 判断 | "政治评论" | 是 |
| `hook_points` | 判断 | ["PR角度", "交易性政治"] | 是 |
| `risk_profile` | 判断 | 8/10, "High sensitivity" | 是 |

*注：图片描述本身也是AI生成，但它是**客观性较强的描述**而非主观判断。

### 4.3 可视化字段映射

```
帖子输入 (Post Input)
  ├─ 文本: title + content
  ├─ 图片: visual elements + text overlays
  └─ 元数据: platform, timestamp, category

            ↓
            
Post Analyzer Skill
  ├─ 提取 Factual Fields (直接copy或简单解析)
  └─ 生成 Judgement Fields (多步骤推理)
      ├─ 语言 & 文化分析
      ├─ 图文联动分析（如果有图片）
      ├─ 勾点识别
      ├─ 风险预判
      └─ 评论策略建议

            ↓
            
结构化输出 (Schema JSON)
  用于 Pattern Extractor 和 Comment Generator 的输入
```

## 5. 工具编排策略 (Tool Orchestration)

### 5.1 为什么用Skill而不是直接Prompt？

**Skill（工作流）** vs **纯Prompt**：

| 维度 | Skill工作流 | 纯Prompt |
|---|---|---|
| **可复用性** | 高。定义一次，多个项目可用 | 低。每次都需重复写 |
| **版本管理** | 清晰。skill版本独立管理 | 混乱。prompt变化难追踪 |
| **失败恢复** | 显性化。步骤级别的fallback | 隐性。只能靠重试 |
| **输出格式** | 严格schema校验。上下游无歧义 | 宽松。容易产生格式混乱 |
| **可审计性** | 每步输入输出都可记录 | 黑盒。难以定位问题 |

我们选择Skill的核心原因：**多步骤复杂流程需要可控性和可审计性**。

### 5.2 工具编排决策树

```
输入帖子
  ↓
[Skill] Post Analyzer
  ├─ 输入：帖子文本 + 图片
  ├─ 步骤1：多模态理解（文字 + 图片）
  ├─ 步骤2：文化语境分析
  ├─ 步骤3：风险预判
  └─ 输出：结构化schema JSON
     ↓
[Skill] Pattern Extractor
  ├─ 输入：Post Analyzer 输出 + 检索到的参考评论
  ├─ 步骤1：参考评论的对噪
  ├─ 步骤2：修辞手法拆解
  ├─ 步骤3：模式抽象
  └─ 输出：通用模式库 + 示例
     ↓
[Skill] Comment Generator
  ├─ 输入：Post Analysis + Pattern Library
  ├─ 步骤1：多风格创作（基于勾点和推荐模式）
  ├─ 步骤2：原创性校验
  ├─ 步骤3：互动预测
  └─ 输出：评论候选 + 最优选择
     ↓
[Hook] Safety Checker
  ├─ 触发：Comment Generator 完成后
  ├─ 校验1：敏感词拦截（正则 + 词库）
  ├─ 校验2：格式校验
  ├─ 校验3：长度校验
  ├─ 校验4：冒犯度预测（可选）
  └─ 结果：PASSED / BLOCKED（阻断不继续）
```

### 5.3 为什么敏感词校验必须做成Hook？

**Hook的优势**：
- **物理阻断**：一旦违反，直接中断管道，不继续发送
- **不依赖LLM自觉**：正则表达式 + 关键词库 = 100%命中（无遗漏）
- **快速失败**：降低无用API调用，节省成本
- **可独立升级**：敏感词库可随时更新，不需重写整个系统

### 5.4 MCP工具接线

**最小权限原则**：
- ✅ 允许：从Reddit获取公开帖子、评论数据（仅读）
- ✅ 允许：调用图像识别API（仅理解，不存储）
- ✅ 允许：查询本地敏感词库（仅检索）
- ❌ 禁止：发布评论、修改数据、访问私人信息
- ❌ 禁止：调用支付API或隐私数据库

详见 `.mcp.json` 配置。

## 6. 全维度风险识别与处理 (Risk Management)

### 6.1 明确的风险分类

**Category A：内容冒犯风险**（最严重）
- 种族/民族刻板印象或贬低表述
- 性别歧视或贬低特定性别
- 宗教或信仰冒犯
- 性取向或性别认同歧视
- **检测**：关键词库 + 语义分析
- **处置**：Hook直接阻断，不生成

**Category B：法律合规风险**
- 人身攻击或名誉伤害
- 隐私泄露（涉及未成年、个人信息）
- 煽动暴力或非法行为
- 知识产权侵犯
- **检测**：关键词库 + 逻辑分析
- **处置**：Hook拦截 + 标注理由

**Category C：平台政策风险**
- 违反特定平台的社区守则
- 垃圾信息或骚扰行为
- 虚假信息传播
- **检测**：平台特定规则库
- **处置**：生成时预警 + Hook拦截

**Category D：战略风险**
- 评论可能激发大规模争议（虽然不违规）
- 容易被曲解或挪作他用
- 可能引发连锁反应
- **检测**：启发式评分
- **处置**：标注风险等级 + 人工审查建议

### 6.2 不可操作区（绝对禁区）

以下内容**绝对禁止生成**，即使经过所有校验仍然要阻断：

1. **针对个体的攻击性评论**
   - 禁止：直接辱骂特定人物（包括公众人物）
   - 禁止：强调某人的生理缺陷或私人信息
   - 允许：对公众人物的政治立场的批判（需中立描述）

2. **系统性压迫的强化**
   - 禁止：强化对边缘群体的刻板印象
   - 禁止：使用历史上用于压迫的语言
   - 允许：指出系统问题并提出改进

3. **涉及未成年人的内容**
   - 禁止：任何涉及儿童的不当内容
   - 禁止：泄露未成年人隐私
   - 禁止：将儿童话题政治化

4. **直接煽动暴力或自伤**
   - 禁止：任何鼓励伤害自己或他人的表述
   - 禁止：提供具体伤害方法
   - 允许：讨论自杀或暴力问题的学术性内容

### 6.3 风险等级与推荐处置

| 等级 | 示例 | 推荐处置 |
|---|---|---|
| **严重(9-10)** | "我觉得[种族A]的人天生低能" | Hook阻断，不生成 |
| **高(7-8)** | "这战争中没有无辜者" | 生成但标注警告，需人工审查 |
| **中等(5-6)** | "这个政策真是脑残" | 生成但标注风险，预先警示 |
| **低(3-4)** | "这个梗烂大街了" | 生成，无需特殊处理 |
| **最小(1-2)** | "这个故事很有意思" | 生成，可直接发布 |

## 7. 多层级失败恢复与降级方案 (Failure Recovery)

### 7.1 API/外部依赖故障

**场景1：Reddit检索API宕机**
```
检索失败
  ↓
[Fallback 1] 使用本地缓存的"通用成功评论模式库"
  ├─ 包含100+条高赞评论的模式提炼
  ├─ 覆盖所有主流话题和风格
  └─ 质量相对稳定但非实时
  ↓
[Fallback 2] 如果缓存也不可用，进入"模式退化模式"
  ├─ 仅使用系统预设的5大通用模式（反转/共情/总结/提问/吐槽）
  ├─ 输出中明确标注"使用预设模式，非基于实时数据"
  └─ 降级生成3条而非5条候选评论
```

**处置代码逻辑**（pseudocode）：
```python
try:
    patterns = search_reddit_patterns(post)
except APIError:
    try:
        patterns = load_cached_patterns()
        log_warning("Using cached patterns")
    except CacheError:
        patterns = DEFAULT_PATTERNS
        log_warning("Using default patterns - degraded mode")
    return generate_comments(patterns, degraded=True)
```

### 7.2 多模态识别故障

**场景2：图片识别失败**
```
图片加载错误
  ↓
[监测] Image Recognition API 返回错误
  ↓
[处置策略]
  ├─ Step 1: 标注Post Analysis输出为 "text-only mode"
  ├─ Step 2: 显性化记录缺失的图片信息
  │   {
  │     "image_status": "FAILED",
  │     "fallback_mode": "text-only",
  │     "warning": "Image recognition failed. Analysis based on text only."
  │   }
  ├─ Step 3: 在Comment Generation中降低对"图文反差"类评论的权重
  └─ Step 4: 输出所有评论时都标注"基于文本分析"的免责声明
```

**绝对禁止**：系统不能凭空编造图片内容或假装理解了图片。

### 7.3 AI生成冒犯内容的应急

**场景3：Comment Generator不小心生成了冒犯内容**
```
评论生成完成
  ↓
[Hook] Safety Checker 捕获敏感词
  ↓
处置流程：
  1. 立即阻断该评论（不进入候选列表）
  2. 记录incident：哪条评论、触发的规则、生成时间
  3. 日志记录供事后分析
  4. 如果超过阈值（同一帖子>2条被拦截），触发警报
  
触发警报后：
  ├─ 暂停该帖子的进一步处理
  ├─ 人工审查生成配置
  ├─ 可能调整Post Analyzer的风险评估
  └─ 重新生成（带更严格的指令）
```

### 7.4 上游分析失败的连锁应急

**场景4：Post Analyzer理解错误（例如：误解了讽刺语气）**
```
后续症状：
  - Pattern Extractor 基于错误理解查找错误的参考模式
  - Comment Generator 生成完全跑题的评论
  - 表现为"评论与帖子风马牛不相及"

检测机制：
  ├─ 检查评论与原帖的"语义相关度"（余弦相似度 > 0.5）
  ├─ 检查风格一致性
  └─ 如果不一致，触发"理解重审"流程

重审流程：
  1. 回滚到 Post Analyzer 阶段
  2. 使用不同的"tone detection"提示词重新分析
  3. 如果新分析与旧分析差异>阈值，标注警告
  4. 基于新分析重新生成评论
```

## 8. 上下文工程规划 (Context Engineering)

### 8.1 常驻上下文（始终保留）

这些信息在系统的整个生命周期中都必须保留：

1. **CLAUDE.md 中的核心规则** (400 tokens)
   - 禁区定义
   - 必填schema字段
   - 验证方式
   - 作用：确保任何阶段都不超界

2. **DESIGN.md 中的核心假设** (300 tokens)
   - H1/H2/H3 三大假设
   - 工具编排思路
   - 作用：指导整体设计方向

3. **敏感词库** (200-300 tokens)
   - 双语禁用词表
   - 作用：Hook的输入源

**总常驻tokens**: ~1000（约占5%）

### 8.2 按需加载上下文（基于请求）

1. **特定帖子的检索结果** (2000-5000 tokens)
   - 加载时机：Pattern Extractor 执行前
   - 卸载时机：Comment Generator 完成后
   
2. **特定帖子的Post Analysis结果** (500-1000 tokens)
   - 加载时机：Pattern Extractor 需要时
   - 卸载时机：评论生成完成
   
3. **平台特定规则库** (300-500 tokens)
   - 加载时机：分析该平台帖子前
   - 卸载时机：切换平台或流程结束

**策略**：使用 "token budget" 跟踪，超限时优先卸载最后加载的上下文。

### 8.3 Token受限时的保留优先级

若总tokens受限（例如：上下文窗口只有4k），优先级顺序：

```
优先级 1（必保留）：
  - CLAUDE.md 核心规则
  - 当前帖子的基本信息（id, title, content）
  - 敏感词库

优先级 2（高保留）：
  - 当前帖子的 hook_points 和 risk_profile
  - Pattern Library（评论生成必需）

优先级 3（可卸载）：
  - 详细的参考评论原文（保留模式摘要即可）
  - 平台特定口语库

优先级 4（最后卸载）：
  - 历史生成记录
  - 详细的文化背景分析
```

## 9. 剩余风险与后续迭代建议

### 9.1 已识别但未完全解决的风险

1. **文化黑箱（Cultural Blind Spots）** - 中等风险
   - **现象**：某些亚文化圈层的梗具有极强的时效性和排他性
   - **示例**：抖音/B站的"非主流"用语、特定粉丝圈的黑话
   - **当前缓解**：通过平台特定的cultural_context字段部分覆盖
   - **后续改进**：
     * 引入"圈层顾问"模式（邀请不同社群成员标注梗的内涵）
     * 建立"梗库"的时效性评分（热度衰减曲线）
     * 定期更新参考评论的"新鲜度"

2. **跨界冒犯（Cross-Cultural Offense）** - 高风险
   - **现象**：在A文化中是无害的梗，在B文化中可能是冒犯
   - **示例**：用词如"人工智能"在某些语境被曲解为贬低
   - **当前缓解**：多语言敏感词库覆盖
   - **后续改进**：
     * 为每个平台和地域建立独立的敏感词库版本
     * 添加"文化上下文检查"专项Skill
     * 建立用户反馈循环，快速发现新的冒犯点

3. **法律责任不清（Legal Gray Area）** - 高风险
   - **问题**：AI生成的评论若被用于侵权行为，谁负责？
   - **现状**：法律尚未完全明确
   - **缓解方案**：
     * 所有生成内容标注"AI Generated"
     * 发布时需人工审查签字
     * 版权归用户，但保留分析记录以防未来审查
     * 考虑添加"免责声明生成"功能

4. **模型滑坡（Model Drift）** - 中等风险
   - **现象**：随着时间推移，参考评论库变得"过时"
   - **缓解**：
     * 定期（月度）更新Reddit参考数据
     * 跟踪"模式有效性指标"（评论实际互动数）
     * 引入在线学习，不断更新模式权重

### 9.2 架构层面的已知限制

1. **纯生成式无法保证原创性**
   - 当前依赖：相似度检测 + 人工审查
   - 后续：考虑添加"重组多个参考模式"的Skill，强制提高多样性

2. **无法预测"真实互动"**
   - 当前代理指标：评论风格匹配度、预测互动类型
   - 限制：无法模拟真实用户行为和时间戳效应
   - 后续：建立A/B测试框架，实时收集反馈

3. **多模态理解依赖外部API**
   - 当前：依赖 Gemini-3-Flash 的视觉能力
   - 风险：API不稳定或费用增加
   - 后续：考虑本地开源模型（如LLaVA）
