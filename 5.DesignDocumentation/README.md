## 🎯 每个问题的核心内容

### C1. 初始假设（Hypotheses H1-H3）

**问题**：你一开始假设什么？

**答案**：
```
H1: "神评论"不是文学创作，而是"语境解码 + 套路应用"
    → 这意味着可教、可学习、可工程化

H2: 多模态理解比纯文本更关键（图片70%的信息）
    → 这意味着必须有图片识别能力

H3: 风险的边界在"有观点但不伤人"
    → 这意味着幽默和冒犯只隔一层纸
```

**为什么重要**：
- H1决定了能否自动化
- H2决定了系统能力的上限
- H3决定了安全策略的严格度

**验证方式**：用Post 001-005验证假设是否成立

### C2. 验证与反证（Verification & Falsification）

**问题**：怎么验证系统是否真的工作？

**答案**：
```
正向验证（系统应该能做到）：
  ✓ Post 001 的评论被我判定为"有效"→ 实际的赞数评估
  ✓ 20条帖子生成80+评论，多样性覆盖
  ✓ 4大对抗场景都有防护方案

反向验证（系统会失败的场景）：
  ✗ 信息完全缺失（无文本、无图）→ 系统降级而不编造
  ✗ Prompt Injection → Hook直接BLOCK
  ✗ 生成冒犯内容 → 4层Hook拦截
  ✗ 模型翻车 → 有应急召回方案
```

**如何做**：
- 正向：运行main.py处理所有20条，对标comments_output.md
- 反向：Post 006是Injection测试，应该看到BLOCKED

### C3. 问题重构（Problem Reframing）

**问题**：本质上在解决什么问题？

**答案**：
```
初始理解：
  "如何写出爆款评论？"
  问题：这是创意问题，AI不擅长

重构为：
  "如何识别高互动评论的底层模式，并复用到新场景？"
  优势：这是模式匹配问题，AI很擅长

进一步重构：
  "如何在保持安全的前提下，应用这些模式？"
  关键：从"创意"变成"控制"
  
最终框架：
  语境解码器（A1） 
    ↓
  参考库学习（A2）
    ↓
  模式应用（A3）
    ↓
  多层防护（Hooks）
```

**为什么重要**：
- 错误的问题定义→错误的解决方案
- 重构后从"不可能"变成"可行"

### C4. Schema设计（Post Analysis）

**问题**：怎么表示对一个帖子的理解？

**答案**：
```
设计思路：
  ├─ 事实字段（客观，可验证）
  │   ├─ id, title, content, platform, images, url
  │   └─ 直接来自输入，无需AI判断
  │
  └─ 判断字段（推理，可审计）
      ├─ core_topic, hook_points, sentiment, risk_level
      └─ AI推理出来，有可追踪源头
```

**为什么分层**：
- 当AI犯错时，能追踪错误在哪一层
- 可信度不同，处理方式也不同
- 便于版本升级和调试

**Schema完整版**：见DESIGN.md 4.1-4.3（14字段）

### C5. 工具编排（Tool Orchestration）

**问题**：为什么选Skill+Hook而不是纯Prompt？

**答案**：
```
Skill的优势：
  ✓ 可复用性：定义一次，多项目用
  ✓ 版本管理：Skill版本独立升级
  ✓ 失败恢复：显性化的fallback
  ✓ 输出格式：严格Schema，无歧义

Hook的优势：
  ✓ 物理阻断：一旦触发直接中断
  ✓ 100%命中：不依赖LLM自觉
  ✓ 快速失败：降低无用API成本
  ✓ 可更新：改敏感词库无需重训

MCP的优势：
  ✓ 权限隔离：工具只做自己的事
  ✓ 降级方案：工具挂了有fallback
  ✓ 审计追踪：每个API调用都记录
```

**对比Prompt only**：
| 特点 | Prompt | Skill+Hook |
|---|---|---|
| 可靠 | ⭐⭐ | ⭐⭐⭐⭐⭐ |
| 可复用 | ⭐ | ⭐⭐⭐⭐⭐ |
| 性能 | 低效 | 快速失败 |
| 可维护 | 难 | 易 |

**决策树**：见DESIGN.md 5.1-5.4

### C6. 风险识别（Risk Management）

**问题**：系统会有什么风险？

**答案**：
```
6大安全分类：
  1. 身份攻击（种族、性别、宗教、年龄）
  2. 人身攻击（辱骂、威胁、贬低）
  3. 暴力（鼓励伤害、自伤、恐怖）
  4. 隐私泄露（个人信息、位置、财务）
  5. 违法内容（毒品、贩运、枪支、诈骗）
  6. 虚假信息（历史否认、医学谎言）

处置方式：
  - 高风险（4-6）：BLOCK（不可操作区）
  - 中风险（2-3）：标记 + 人工审查
  - 低风险（0-1）：允许发布
```

**如何防护**：
- Layer 1: 关键词库（500+词）
- Layer 2: 结构校验（完整性）
- Layer 3: NLP评分（语义分析）
- Layer 4: 相似度检查（不抄袭）

### C7. 失败恢复（Failure Recovery）

**问题**：什么会出错？怎么应对？

**答案**：
```
场景1：API宕机（Reddit搜索不可用）
  降级：用本地缓存或通用套路库

场景2：图片识别失败
  降级：用图片alt text或标记为TEXT_ONLY_MODE

场景3：AI生成冒犯内容
  恢复：Hook在Layer 3拦截 → 不输出给用户

场景4：上游分析失败
  恢复：标记为INCOMPLETE → 建议用户补充信息
```

**原则**：
- 宁可降级输出，也不编造内容
- 降级要显性标注（用户知道限制）
- 有retry机制（不是一次性失败）

### C8. 上下文工程（Context Engineering）

**问题**：面对Token限制，怎么优化？

**答案**：
```
常驻上下文（始终保留）：
  • CLAUDE.md的核心规则（Rule 1-5）
  • 当前Skill的输入/输出定义
  • 最常见的错误示例

按需加载（运行时）：
  • 特定帖子的A1分析
  • 参考库的相关评论
  • 历史的错误案例（如果重复错误）

优先级保留：
  受限时保留：
    P1: 规则和禁区
    P2: 当前任务的输入/输出
    P3: 详细的参考案例
    P4: 日志和历史
```

**Token预算**：
- Prompt系统部分：2000-3000 tokens
- 用户输入+当前任务：1000-2000 tokens
- 参考案例+上下文：1000-3000 tokens
- 保留用于输出：1000+ tokens

### C9. 剩余风险（Residual Risks）

**问题**：什么我们还没有解决？

**答案**：
```
P0（立即改进）：
  1. 敏感词库需要更新（隔周）
  2. 模型容易生成"保险但无趣"的评论
  3. 没有学习新的流行梗（依赖人工更新参考库）

P1（近期优化）：
  4. 即时性弱（参考库有延迟）
  5. 缺乏个人风格（系统是"通用拼接"）
  6. 多语言支持有限（仅中英）

P2（长期方向）：
  7. 无用户反馈循环（不知道实际表现如何）
  8. 无法处理极端边界场景（组合风险）
```

**改进方向**：
- P0: 定期更新敏感词库 + 调整套路权重
- P1: 接入实时趋势API + 用户风格参数化
- P2: 建立反馈闭环 + 多语言扩展
